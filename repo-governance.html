Technische Governance

Veröffentlichungsregeln für die gültige Fassung.

Diese Seite beschreibt, wie Notariat8 seine GitHub-Ablagen schützt. Die für die Veröffentlichung freigegebene Hauptfassung darf nicht rückwirkend überschrieben oder gelöscht werden. Weitere Freigabepflichten werden erst verbindlich, wenn die jeweilige technische Prüfung stabil läuft und als Regel dokumentiert ist.

Geltende Grundregel

Für die veröffentlichten Ablagen gilt: Die freigegebene Hauptfassung bleibt erhalten. Sie darf nicht rückwirkend überschrieben oder gelöscht werden. Eine Prüfung als Veröffentlichungsbedingung oder eine verpflichtende Vier-Augen-Freigabe wird erst aktiviert, wenn die Prüfung verlässlich eingerichtet ist und die Regel für diese Ablage dokumentiert wurde.

Arbeitsablauf

Änderungen werden vor Veröffentlichung geprüft, mit einer nachvollziehbaren Änderungsnotiz gespeichert und in GitHub sichtbar. Für produktionskritische Änderungen wird eine Vier-Augen-Freigabe über einen Änderungsvorschlag mit Prüfung und Freigabe, auf GitHub Pull Request genannt, bevorzugt. Bei dringenden Korrekturen an dieser statischen Website bleibt eine direkte Veröffentlichung zulässig, damit Fehler schnell behoben werden können; die Änderung bleibt trotzdem nachvollziehbar.

Demo-Evidenz

Vorführstände werden über geschützte Änderungsvorschläge, statische Dokumente und automatisierte Tests nachvollziehbar gemacht. Diese Nachweise enthalten keine Mandatsdaten und lösen keine Cloud-Aktion aus; sie belegen nur, welcher sichtbare Stand geprüft wurde.

Softwarebestandteile

Wenn für eine freigegebene Fassung eine Software Bill of Materials (SBOM), deutsch Software-Stückliste, geführt und geprüft wird, unterstützt sie die Zuordnung von Open-Source- und Drittanbieter-Komponenten, Versionen, Abhängigkeiten, Sicherheitsmeldungen und Lizenzfragen zu genau dieser Fassung. Sie ersetzt keine eigene Sicherheits- oder Rechtsprüfung.

Prüfungen als Veröffentlichungsbedingung

Sobald eine Ablage eine stabile automatisierte Prüfung hat, kann diese Prüfung als Veröffentlichungsbedingung hinterlegt werden. Dann lässt GitHub eine Änderung erst zu, wenn die Prüfung erfolgreich war. Der Name der Prüfung muss eindeutig sein, damit eine Freigabe nicht durch unklare technische Bezeichnungen blockiert wird.
